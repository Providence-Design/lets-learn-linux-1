### Q1 — Where am I, who am I, what system am I on?

**Commands run:**

```bash
pwd
whoami
uname -a
```

**Sample outputs:**

```
/home/aba

aba

Linux cyphercore-dev 6.1.0-21-amd64 #1 SMP PREEMPT_DYNAMIC Debian 6.1.90-1 (2024-05-03) x86_64 GNU/Linux
```

**What each command tells you:**

- **`pwd`** (print working directory) shows the absolute path of your current location in the filesystem. On an unknown server this immediately tells you whether you landed in a home directory, a service account's directory, or somewhere unexpected like `/tmp` or `/`.

- **`whoami`** prints the effective username of the current session. This tells you whether you are a regular user, a service account (e.g. `www-data`, `postgres`), or `root`. The answer changes what you can and cannot do.

- **`uname -a`** — prints all kernel and system information: hostname, kernel version, build date, CPU architecture. The `-a` flag means "all".

**Why a DevSecOps engineer runs these three first when SSHing into an unknown server:**

The moment you connect to an unfamiliar server you need a threat picture before you touch anything. `pwd` and `whoami` together tell you your footprint — are you root, are you in a sensitive directory? `uname -a` gives you the exact kernel version, which you then cross-reference against the CVE database. A server running Linux kernel `5.15.0-78` might be vulnerable to a local privilege escalation like CVE-2023-32629 (GameOver(lay)) that affects Ubuntu kernels in that range. If you don't run `uname -a` first, you could spend hours hardening application configs on a box that an attacker can root in seconds via a known kernel exploit. These three commands cost five seconds and tell you whether the server is fundamentally safe to work on.

### Q2 — The Linux Filesystem Root

**Command run:**

```bash
ls /
```

**Sample output:**

```
bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
```

**Five directories explained:**

| Directory | Purpose |
|-----------|---------|
| `/etc` | System-wide configuration files. Everything from SSH daemon settings (`/etc/ssh/sshd_config`) to user account info (`/etc/passwd`) lives here. A DevSecOps engineer checks this directory when auditing a server. |
| `/var` | Variable data files that grow and change at runtime. Logs (`/var/log`), mail spools, package caches. Log monitoring tools point here. |
| `/proc` | A virtual filesystem exposing the kernel's view of running processes and hardware. Files here are not on disk — they are generated by the kernel in real time. `/proc/1/cmdline` shows the command that started PID 1. |
| `/home` | Home directories for regular users. Each user gets a subdirectory (e.g. `/home/aba`). On a server with many engineers, this is where personal config files and scripts accumulate. |
| `/tmp` | World-writable temporary storage. Cleared on reboot. Used by applications for scratch files. Also commonly abused by attackers to stage scripts and binaries because anyone can write here. |

**What is the Filesystem Hierarchy Standard (FHS)?**

The FHS is a specification maintained by the Linux Foundation that defines where different types of files must live on a Linux system. It answers questions like: where do executables go (`/bin`, `/usr/bin`)? Where do logs go (`/var/log`)? Where do config files go (`/etc`)?

**Why does it exist?**

Without FHS, every Linux distribution would organise files differently. A script written on Ubuntu that reads `/etc/nginx/nginx.conf` would break on a custom system that put configs in `/usr/local/conf/`. Package managers, deployment tools, monitoring agents, and log shippers all hardcode paths because FHS guarantees they will be correct.

**What breaks on a shared production server without it?**

- Log agents like Filebeat or Fluentd fail to find log files.
- Monitoring tools like Nagios or Prometheus exporters cannot locate service binaries.
- Automation scripts (Ansible playbooks, shell scripts) fail silently or destructively when paths don't exist where expected.
- Incident responders cannot triage quickly because nothing is where they expect it to be in a 2 AM outage, that costs minutes that matter.

### Q3 — Seeing Hidden Files

**Commands run:**

```bash
ls ~
ls -la ~
```

**What is different between the two outputs?**

`ls ~` shows only the visible files and directories in the home directory — for a fresh user this might be just `projects/` or nothing at all. `ls -la ~` adds two flags: `-l` gives the long format (permissions, link count, owner, group, size, timestamp) and `-a` reveals all files including those whose names begin with a dot (`.`). The second command typically shows many more entries: `.bashrc`, `.bash_history`, `.profile`, `.ssh/`, and others.

**What makes a file hidden in Linux?**

A file is hidden simply by naming it with a leading dot — for example `.hidden_file`. There is nothing special about the file type, the filesystem entry, or the inode. It is purely a naming convention that `ls` respects by default. Any file or directory can be "hidden" this way and "unhidden" by renaming it. It is a display convention, not a security feature.

**Two real hidden files in a home directory:**

1. **`.bashrc`** — A shell script sourced every time an interactive non-login Bash session starts. It contains user-specific aliases (e.g. `alias ll='ls -la'`), environment variables, and PATH additions. Attackers who gain shell access to a server often modify `.bashrc` to establish persistence — adding a reverse shell or a malicious alias that runs every time the user opens a terminal.

2. **`.ssh/`** A directory containing SSH keys and the `authorized_keys` file. `authorized_keys` lists public keys that are permitted to authenticate as this user without a password. An attacker who gains write access to this file can add their own public key and log in at will — which is why its permissions must be exactly `600` and the `.ssh/` directory must be `700`.

### Q4 — Create the Directory Structure

**Command run:**

```bash
mkdir -p ~/projects/cyphercore/{configs,logs/{access,errors,archive},reports}
```

**Verification:**

```bash
tree ~/projects/cyphercore
```

**Expected tree output:**

```
/home/aba/projects/cyphercore
├── configs
├── logs
│   ├── access
│   ├── archive
│   └── errors
└── reports

6 directories, 0 files
```

**Which flag was used and what does it do?**

The `-p` flag (short for `--parents`) tells `mkdir` two things: first, create all intermediate parent directories that don't yet exist (so `~/projects/cyphercore/` is created automatically even though `~/projects/` didn't exist); second, do not throw an error if the target directory already exists. Without `-p`, you would have to create each level one at a time, and running the command twice would error out on the second run.

**What is brace expansion and how did it help?**

Brace expansion is a Bash feature that generates multiple strings from a pattern with commas inside `{}`. The shell expands `{configs,logs,reports}` into three separate words before the command even runs. Nesting `logs/{access,errors,archive}` generates `logs/access`, `logs/errors`, and `logs/archive` in one expression. The single command above expands to effectively seven `mkdir` calls without writing them all out. It is not `mkdir` that understands the braces it is Bash itself, happening before any command executes.

**Could you do this without brace expansion — and would you want to?**

Yes, you could run seven separate `mkdir -p` commands. You would not want to because it is slow, error-prone (typos in repeated paths), harder to read in a script, and impossible to review at a glance to confirm the full structure is correct. Brace expansion makes the intent explicit and the structure visible in a single line.

### Q5 — Create Placeholder Files

**Command run (from `~/projects/cyphercore/`):**

```bash
touch configs/app.conf configs/db.conf logs/access/access.log logs/errors/error.log reports/weekly_report.txt
```

**What does `touch` actually do beyond creating files?**

`touch` has two distinct jobs. Its primary purpose is to update the access timestamp (`atime`) and modification timestamp (`mtime`) of an existing file to the current time. Creating a new file is a side effect that happens when the target path does not yet exist. So `touch` is more accurately described as a timestamp updater that also creates files when needed.

**What happens if you run it on an existing file?**

Nothing destructive. The file's contents are completely unchanged. Only the timestamps are updated. Running `ls -l` before and after shows that the modification time changes to the current moment. The file size and permissions remain identical.

**Why does this behaviour matter in automation scripts?**

Many automation tools Make, Ansible, deployment pipelines — use timestamps to decide whether a file needs to be processed. A script might check: "is this config file newer than the last deployment?" If you use `touch` to signal that a file has been reviewed or updated without changing its contents, it triggers the downstream process. More critically, if scripts check for the existence of placeholder files before proceeding (`if [ -f logs/access/access.log ]; then ...`), a missing placeholder causes a silent failure. Creating them at project setup prevents this entire class of bug.

### Q6 — Long Listing of configs/

**Command run:**

```bash
ls -lh ~/projects/cyphercore/configs/
```

**Sample output:**

```
total 0
-rw-r--r-- 1 aba aba 0 Jun  2 09:35 app.conf
-rw-r--r-- 1 aba aba 0 Jun  2 09:35 db.conf
```

**What do the file sizes tell you?**

Both files are 0 bytes. They were created with `touch` which does not write any content. This is expected they are placeholders confirming the paths exist so downstream scripts don't fail on a missing file check. The `0` under size is the confirmation.

**Breaking down the permission string `-rw-r--r--`:**

```
- rw- r-- r--
│  │   │   └── Others: read only (r=4, w=0, x=0) → octal 4
│  │   └────── Group:  read only (r=4, w=0, x=0) → octal 4
│  └────────── Owner:  read and write (r=4, w=2, x=0) → octal 6
└───────────── File type: - means regular file (d=directory, l=symlink)
```

Full octal representation: `644`. The owner can read and modify the file. Everyone else can only read it. No one has execute permission — correct for a config file, which is data, not a program.

**What does the `-h` flag add?**

`-h` (human-readable) formats file sizes in units that are easy to read — bytes (B), kilobytes (K), megabytes (M), gigabytes (G) — rather than always showing raw bytes. For a 2,147,483,648-byte file, `-h` shows `2.0G`. Without it you get the number `2147483648` which is harder to parse at a glance. In a directory with hundreds of log files of various sizes, `-h` makes capacity issues obvious immediately.

### Q7 — Full Tree Verification

**Command run:**

```bash
tree ~/projects/cyphercore
```

**Expected output:**

```
/home/aba/projects/cyphercore
├── configs
│   ├── app.conf
│   └── db.conf
├── logs
│   ├── access
│   │   └── access.log
│   ├── archive
│   └── errors
│       └── error.log
└── reports
    └── weekly_report.txt

6 directories, 5 files
```

**How is `tree` different from `ls -R`?**

`ls -R` recursively lists directories but outputs them as a flat sequence of blocks — each directory header followed by its contents. It is hard to visualise the nesting relationship between directories. `tree` draws the hierarchy visually using box-drawing characters, showing parent-child relationships at a glance. At five levels deep, `ls -R` becomes nearly unreadable; `tree` remains clear.

**When would a DevSecOps engineer run `tree` on a live server?**

- **Deployment validation**: Confirming a deployment script created the exact directory structure a service expects — missing a subdirectory can cause an application to crash on first write.
- **Security audit**: Checking for unexpected directories or files in a web root or home directory. A `tree /var/www/html` that reveals a `shell.php` or an unexpected `uploads/backdoor/` is an immediate red flag.
- **Incident response**: When investigating a compromised server, `tree /tmp` and `tree /dev/shm` quickly reveal staged attacker tools and scripts without having to navigate directory by directory.

### Q8 — Writing Log Lines with echo

**Commands run:**

```bash
echo "2025-06-02 08:14:33 INFO  Application started successfully" >> logs/access/access.log
echo "2025-06-02 08:14:55 WARN  High memory usage detected: 87%" >> logs/access/access.log
echo "2025-06-02 08:15:10 ERROR Database connection timeout — retrying (attempt 1/3)" >> logs/access/access.log
cat logs/access/access.log
```

**`>` vs `>>`:**

| Operator | Behaviour |
|----------|-----------|
| `>` | Redirect stdout to a file. **Truncates** the file first — all existing content is destroyed before writing. If the file doesn't exist, it is created. |
| `>>` | Redirect stdout and **append** to the file. Existing content is preserved. The new content is added after the last line. If the file doesn't exist, it is created. |

**Proof — what happens with `>`:**

Running `echo "2025-06-02 08:14:33 INFO Application started successfully" > logs/access/access.log` after the other two lines are already in the file would destroy the WARN and ERROR lines entirely, leaving only the INFO line. This is because `>` opens the file with the `O_TRUNC` flag, zeroing it before writing. `cat logs/access/access.log` would show only one line.

Restoring the file means re-running all three `echo >>` commands.

**Why confusing `>` and `>>` is dangerous in a production log rotation script:**

Log rotation scripts are typically run on a schedule (cron, systemd timer) and operate on live log files that other services are actively writing to. Using `>` where `>>` is intended or accidentally dropping an `>` character silently destroys the existing log content before the rotation completes. You lose the audit trail for the past rotation period: access records, error events, security events. In a regulated environment (PCI-DSS, HIPAA) this is a compliance failure. In an incident investigation it means the evidence you need for forensics is gone. The command succeeds, returns exit code 0, and produces no error message. The damage is invisible until someone tries to read the logs.

### Q9 — Reading the Log Two Ways

**Commands run:**

```bash
cat logs/access/access.log
tac logs/access/access.log
```

**Difference:**

`cat` (concatenate) reads the file from the first byte to the last and prints it top to bottom — the oldest log entry at the top, the most recent at the bottom. `tac` is `cat` spelled backwards and does the opposite: it reads the file and outputs lines in reverse order, so the most recent log entry appears first.

**Which to reach for in an 80,000-line incident log?**

`tac` first, then pipe it into `head` or a pager. The most recent events are at the bottom of a log, and in an incident you want to see what just happened not scroll through 79,990 lines of history to get there. `tac logs/access/access.log | head -50` immediately shows the last 50 events in reverse-chronological order.

**Third command that shows only the last 10 lines:**

```bash
tail logs/access/access.log
```

`tail` by default outputs the last 10 lines of a file. With `-f` (follow) it keeps the file open and streams new lines as they are written essential for watching a live log during an incident: `tail -f logs/access/access.log`.

### Q10 — Writing the Error Log with a Heredoc

**Command run:**

```bash
cat >> logs/errors/error.log << EOF
2025-06-02 08:15:10 ERROR    Database connection timeout — retrying (attempt 1/3)
2025-06-02 08:15:13 ERROR    Database connection timeout — retrying (attempt 2/3)
2025-06-02 08:15:16 CRITICAL Database connection failed — all retries exhausted
2025-06-02 08:15:16 CRITICAL Triggering failover to secondary DB at 10.0.0.52
EOF
```

**Reading it back:**

```bash
cat logs/errors/error.log
```

---

**What is a heredoc and what does it solve?**

A heredoc (here document) is a Bash construct that allows you to supply a multi-line block of text as stdin to a command, without creating a temporary file or running the command multiple times. The syntax is `<< DELIMITER` everything typed until the delimiter appears alone on a line is fed to the command as standard input. It solves the problem of writing four (or forty) lines of structured content in a single atomic operation, which is both readable and scriptable. Using four separate `echo >>` commands means four separate write system calls and four opportunities for a typo or interruption to leave the file in an inconsistent state.

---

**`'EOF'` (quoted) vs `EOF` (unquoted):**

```bash
DBNAME=cyphercore

# Unquoted — shell expands variables inside the heredoc
cat << EOF
The database name is $DBNAME
EOF
# Output: The database name is cyphercore

# Quoted — shell treats everything literally, no expansion
cat << 'EOF'
The database name is $DBNAME
EOF
# Output: The database name is $DBNAME
```

When the delimiter is **unquoted** (`EOF`), Bash performs variable expansion, command substitution, and arithmetic expansion on the heredoc body — exactly as it would in a double-quoted string. When the delimiter is **quoted** (`'EOF'` or `"EOF"` or `\EOF`), Bash treats the entire heredoc body as a raw literal string and makes no substitutions. The dollar sign, backticks, and `$()` are all passed through unchanged.

This distinction matters enormously. If you are writing a script template that will itself be a shell script (i.e., the output should contain `$VARIABLE` literally for later execution), you must quote the delimiter, otherwise Bash expands your template variables immediately and writes garbage values or empty strings into the output file.

---

**Why might someone on a compromised server prefer heredoc over opening nano or vim?**

Interactive editors like `nano` and `vim` leave forensic traces: they write swap files (`/tmp/.error.log.swp`), they appear in `history`, and on some systems they log to `wtmp`/`utmp`. A heredoc embedded in a shell command writes content in a single process invocation it shows in `history` as one line (which an attacker who clears `.bash_history` can remove in one operation) and creates no swap file artifacts on disk. For an attacker trying to minimise their filesystem footprint while staging a script, a heredoc piped through `bash` is preferable. For a defender doing forensics, this is why you examine `/proc/<pid>/fd/` for file descriptors and check `auditd` logs for `open()` syscalls, not just swap files.

### Q11 — Pager Commands

**Commands run:**

```bash
more logs/errors/error.log
less logs/errors/error.log
```

**Key differences:**

| Feature | `more` | `less` |
|---------|--------|--------|
| Navigation | Forward only (space to advance) | Forward and backward (arrow keys, PgUp/PgDn) |
| Memory | Loads entire file before displaying | Streams — loads only what is displayed |
| Search | Limited | Full regex search with `/` |
| Speed on large files | Slow (must read whole file first) | Fast (starts immediately) |
| Quit | `q` or reaching end of file | `q` |

**Which loads the entire file into memory first?**

`more` reads the entire file before displaying it. `less` is a streaming pager it reads only what it needs to display the current screen, making it start almost instantly regardless of file size.

**On a server with 512MB RAM and a 3GB log file — which do you use?**

`less` without hesitation. Loading a 3GB file into `more` would consume all available RAM and cause the server to start swapping, potentially degrading running services in the process. `less` never holds more than a screen's worth of data in memory.

**Three keyboard shortcuts for `less`:**

| Action | Shortcut |
|--------|----------|
| Search for a pattern | `/pattern` then Enter — press `n` for next match, `N` for previous |
| Jump to end of file | `G` (capital G) |
| Quit | `q` |

Bonus: `F` (capital F) puts `less` into follow mode equivalent to `tail -f` streaming new content as it is written. Useful for watching a live log without leaving the pager.

### Q12 — Archive and Rename

**Commands run:**

```bash
cp logs/errors/error.log logs/archive/error_2025-06-02.log
mv logs/access/access.log logs/access/access_2025-06-02.log
ls -l logs/access/ logs/archive/
```

**Fundamental difference between `cp` and `mv`:**

`cp` creates a new copy of the file at the destination path. After `cp`, both the source and destination exist independently they have separate inodes and separate data blocks (on the same filesystem, the data is physically duplicated). `mv` does two things depending on context: if source and destination are on the same filesystem, it only updates the directory entry (the filename pointer) — no data is copied, the inode number stays the same, and the operation completes in microseconds regardless of file size. If source and destination are on different filesystems, `mv` copies the data then deletes the source.

**After renaming — does the original filename still exist?**

No. `mv logs/access/access.log logs/access/access_2025-06-02.log` removes the name `access.log` from the directory and creates the name `access_2025-06-02.log` pointing to the same inode. The original name is gone. `ls logs/access/` will show only `access_2025-06-02.log`.

**Does copying a large file duplicate all data on disk immediately?**

On a standard filesystem (ext4, XFS) with a standard `cp`, yes a full copy of the data blocks is written immediately. A 2GB error log becomes 4GB of disk usage after copying. Some filesystems support copy-on-write (CoW) semantics (Btrfs, ZFS) where `cp --reflink=auto` creates a logical copy that shares data blocks until one of the copies is modified. On ext4 (the most common production filesystem), there is no reflink support and the full copy is written immediately.

### Q13 — Remove Directories

**Commands run:**

```bash
mkdir reports/temp_drafts
rmdir reports/temp_drafts
rmdir logs/
```

**What happened when you tried to remove `logs/`?**

The command failed with:
```
rmdir: failed to remove 'logs/': Directory not empty
```

`rmdir` only removes empty directories. `logs/` contains `access/`, `errors/`, and `archive/` subdirectories (with files inside them), so `rmdir` refuses to proceed.

**`rmdir` vs `rm -r`:**

`rmdir` removes a single empty directory and fails safely if the directory has any contents. `rm -r` (recursive) removes a directory and everything inside it all subdirectories and all files without stopping for non-empty directories. It is a complete deletion with no recovery.

**What makes `rm -rf` dangerous and what rule should every engineer follow?**

`rm -rf` combines recursive deletion (`-r`) with force (`-f`), which suppresses all confirmation prompts and ignores non-existent file errors. A misplaced space can be catastrophic: `rm -rf /home /aba` deletes the entire `/home` directory tree rather than `/home/aba`. Running it as root on `/` has bricked production servers.

The rule: **always double-check the path, never run `rm -rf` on a variable without quoting and validating it, and always confirm you are on the correct server before running any destructive command.** Many engineers alias `rm` to `rm -i` (interactive, asks for confirmation) in non-automated contexts. In scripts, prefer moving files to a dated trash directory and deleting the trash on a schedule, so mistakes are recoverable.

### Q15 — Hard Links and Inode Numbers

**Commands run:**

```bash
printf "DB_HOST=10.0.0.10\nDB_PORT=5432\nDB_NAME=cyphercore_prod\n" > configs/db.conf
ln configs/db.conf configs/db_hardlink.conf
ls -li configs/
```

**Sample output:**

```
total 8
524291 -rw-r--r-- 2 aba aba 47 Jun  2 14:03 app.conf
524289 -rw-r--r-- 2 aba aba 47 Jun  2 14:05 db.conf
524289 -rw-r--r-- 2 aba aba 47 Jun  2 14:05 db_hardlink.conf
```

**What do you notice about the inode numbers of both files?**

`db.conf` and `db_hardlink.conf` share the **same inode number** (e.g. `524289`). This is the definitive proof that they are not two separate files — they are two names pointing to the same underlying data structure on disk.

**What does the link count (third column) represent?**

The number `2` in the third column is the hard link count the number of directory entries that point to this inode. Before creating the hard link, `db.conf` had a link count of `1`. After `ln`, the inode's link count incremented to `2`. This counter is maintained by the filesystem in the inode itself. When the link count reaches `0`, the filesystem knows no name points to this inode and the data blocks are freed.

**What is an inode in your own words?**

An inode (index node) is a data structure stored in the filesystem that holds all metadata about a file except its name. It records: file type, permissions, owner UID and GID, file size, timestamps (created, accessed, modified), and the addresses of the data blocks on disk that contain the file's actual content. A filename in a directory is just a human-readable label that maps to an inode number — like a database record ID. One inode can have many names (hard links) pointing to it. The kernel uses the inode number, not the filename, to access file data.

### Q16 — Deleting the Original and Reading the Hard Link

**Commands run:**

```bash
rm configs/db.conf
cat configs/db_hardlink.conf
ls -li configs/
```

**Sample output:**

```
DB_HOST=10.0.0.10
DB_PORT=5432
DB_NAME=cyphercore_prod

524289 -rw-r--r-- 1 aba aba 47 Jun  2 14:05 db_hardlink.conf
```

**Is the data still there?**

Yes, completely. The content of `db.conf` is fully readable through `db_hardlink.conf`.

**Explain why using inodes and link counts:**

`rm configs/db.conf` did not delete the data it deleted the directory entry (the name `db.conf`) and decremented the inode's link count from `2` to `1`. The inode `524289` still exists because its link count is greater than zero. The filesystem only marks data blocks as available for reuse when the link count drops to `0` and no process has the file open. Since `db_hardlink.conf` still points to inode `524289`, the data is intact and fully accessible.

**What would need to happen for the data to be permanently deleted?**

`rm configs/db_hardlink.conf` would remove the last directory entry pointing to inode `524289`, decrementing the link count to `0`. With no names pointing to it and no processes with the file open, the filesystem would mark the inode as free and add its data blocks back to the free block pool. The data is not immediately overwritten, but it is recoverable only with forensic tools until those blocks are reused.

**Link count after deletion:**

The link count on `db_hardlink.conf` dropped from `2` to `1` visible in `ls -li`. This is the filesystem confirming that only one name now references this inode.

**How could an attacker use hard links to hide data?**

An attacker could create a hard link to a malicious file from an innocuous-looking path (e.g. linking a keylogger binary to `/var/log/.syslog_helper`) and then delete the original conspicuous path. A search for the original filename finds nothing. A `find` by name finds nothing. Only `find / -inum <inode_number>` or tools that scan all directory entries for duplicate inodes would reveal the hidden name. This technique is used to hide tools and maintain persistence while evading simple file-existence checks. Defenders counter it with inode-aware forensic tools like `fls` (from The Sleuth Kit) that list all directory entries including deleted ones.

### Q17 — Symbolic Links

**Commands run:**

```bash
echo "APP_PORT=8080\nAPP_ENV=staging\nAPP_DEBUG=false" > configs/app.conf
ln -s configs/app.conf ~/projects/cyphercore/app_config_link
ls -la ~/projects/cyphercore/
rm configs/app.conf
cat ~/projects/cyphercore/app_config_link
```

**Listing output (relevant line):**

```
lrwxrwxrwx 1 aba aba 15 Jun  2 14:22 app_config_link -> configs/app.conf
```

**Failed read output:**

```
cat: ~/projects/cyphercore/app_config_link: No such file or directory
```

**What character identifies a symbolic link?**

The letter `l` at the start of the permission string (`lrwxrwxrwx`). The listing also shows the link target after `->`: `app_config_link -> configs/app.conf`. The permissions on a symlink itself (`rwxrwxrwx`) are irrelevant — access is controlled by the permissions on the target file.

**What happened when you read the broken link — what is this called?**

The read failed with "No such file or directory" even though `app_config_link` visibly exists in the directory listing. This is called a **dangling symlink** (or broken symlink). The symlink file itself still exists and points to the path `configs/app.conf`, but that target path no longer resolves to anything — the target was deleted. The kernel follows the symlink to the target path and finds nothing there, so it reports the error as if the symlink itself doesn't exist.

**Why does a dangling symlink in a deployment pipeline cause hard-to-debug failures?**

A deployment script might check `if [ -f app_config_link ]`  but `[ -f ]` follows symlinks and returns false for a dangling link, even though `ls` shows the symlink plainly. The script fails with a cryptic "file not found" error that points to the symlink, not to the missing target. The engineer on call sees the symlink in the directory and thinks "but it's right there" they don't immediately think to check whether the *target* exists. At 2 AM under pressure, this takes minutes to diagnose. Meanwhile, `[ -L ]` tests whether a symlink exists without following it but most scripts don't use this. The root cause (missing target) and the visible symptom (symlink present) are in different locations, which is exactly the kind of mismatch that turns a 5-minute fix into a 45-minute outage.

### Q18 — Hard Link vs Soft Link Comparison Table

| Property | Hard Link | Soft Link |
|----------|-----------|-----------|
| Shares inode with original? | **Yes** — same inode number | **No** — symlink has its own inode; stores path to target |
| Works across different filesystems? | **No** — inode numbers are unique per filesystem | **Yes** — stores a path string, which can cross filesystem boundaries |
| Survives deletion of original? | **Yes** — data persists until link count reaches 0 | **No** — becomes a dangling symlink; target data is gone |
| Can link to a directory? | **No** (root only, not allowed on most systems to prevent loops) | **Yes** — symlinks to directories are common |
| Shows as `l` in `ls -la`? | **No** — appears identical to a regular file | **Yes** — file type character is `l` |
| Detectable by matching inodes in `ls -li`? | **Yes** — duplicate inode numbers reveal hard links | **No** — symlink has its own unique inode |

**Real-server use case — Hard Link (DevSecOps context):**

A compliance team requires that audit logs be stored in two locations: `/var/log/app/audit.log` (for the application) and `/var/log/archive/audit.log` (for the SIEM ingestion agent). Using a hard link instead of copying means both paths point to the same data there is no risk of the two copies diverging, no extra disk usage, and writes to either path update the same file. When the SIEM agent reads `/var/log/archive/audit.log`, it always gets the current data, even if the application continues writing to `/var/log/app/audit.log`.

**Real-server use case — Soft Link (DevSecOps context):**

An Nginx web server has its configuration at `/etc/nginx/sites-available/cyphercore.conf`. To enable the site, the convention is to create a symlink: `ln -s /etc/nginx/sites-available/cyphercore.conf /etc/nginx/sites-enabled/cyphercore.conf`. This is the standard Nginx pattern — the master config is in `sites-available`, and `sites-enabled` contains only symlinks to active configs. Disabling a site is `rm /etc/nginx/sites-enabled/cyphercore.conf` (the master config is untouched). This pattern is used by Apache, Nginx, systemd unit files, and many other tools because symlinks make enable/disable operations safe and reversible.

### Q19 — Separating stdout and stderr

**Command run:**

```bash
ls logs/ nonexistent_directory/ 1>reports/stdout_output.txt 2>reports/stderr_output.txt
cat reports/stdout_output.txt
cat reports/stderr_output.txt
```

**stdout_output.txt contents:**

```
logs/:
access
archive
errors
```

**stderr_output.txt contents:**

```
ls: cannot access 'nonexistent_directory/': No such file or directory
```

**Which content landed where and why?**

The directory listing for `logs/` went to `stdout_output.txt` because it is normal program output the successful result of the command. The error message went to `stderr_output.txt` because `ls` writes error conditions to stderr (stream 2), not stdout (stream 1). The shell independently redirected each stream to its respective file. The terminal showed nothing because both streams were redirected away from the screen.

**All three streams, their numbers, and what each carries:**

| Stream | Number | Carries |
|--------|--------|---------|
| stdin | 0 | Input fed to a process (keyboard, pipe, file) |
| stdout | 1 | Normal program output — results, data, success messages |
| stderr | 2 | Error messages, warnings, diagnostic output |

**What does `2>` mean specifically?**

`2>` redirects file descriptor 2 (stderr) to a file. It is syntactically identical to `1>` (or just `>`), but targets the error stream instead of the output stream. `2>/dev/null` is the common pattern for silencing errors it redirects them to the null device where they are discarded.

**`2>&1` explained:**

```bash
ls logs/ nonexistent_directory/ 2>&1 | cat
```

`2>&1` means "redirect file descriptor 2 to wherever file descriptor 1 is currently pointing." At the moment this is evaluated, fd 1 is going to the pipe. So `2>&1` makes stderr also go into the pipe both streams are merged and `cat` receives everything. Without `2>&1`, the pipe only carries stdout and stderr bleeds onto the terminal as unsorted noise.

**Real deployment script use case:**

```bash
./deploy.sh 2>&1 | tee -a /var/log/deploy.log
```

This runs the deploy script, merges both streams, and uses `tee` to simultaneously display output on the terminal and append it to a log file. Without `2>&1`, error messages from the deployment script would appear on the terminal but not in the log file the log would show a successful-looking run with no errors, and the error would only exist in the operator's memory. In a CI/CD pipeline where the terminal session disappears after the run, those errors are lost permanently.

### Q20 — Writing the Weekly Report with Heredoc

**Command run:**

```bash
cat > reports/weekly_report.txt << 'EOF'
========================================
WEEKLY SECURITY REPORT — CypherCore Systems
Week Ending: 2025-06-06
Prepared By: aba Mensah
========================================

SUMMARY:
No critical incidents recorded this week.
One WARN: memory usage spike at 08:14 on 2025-06-02.
One CRITICAL: DB connection failure at 08:15 on 2025-06-02.
Failover to secondary DB executed successfully.

RECOMMENDED ACTION:
Review DB connection pool settings before next deployment.
========================================
EOF
cat reports/weekly_report.txt
```

**Did you quote the delimiter — and why does that choice matter?**

Yes, the delimiter is quoted (`'EOF'`). The report content contains no variables that should be expanded, but the `—` em-dashes and the general structure suggest this is meant to be treated as a literal document. Using a quoted delimiter guarantees the output is byte-for-byte identical to what was typed, regardless of any shell variables that happen to be set. If an unquoted delimiter were used and a variable like `$ANALYST` or `$WEEK` happened to be set in the environment, those strings would silently expand potentially inserting wrong values into an official report without any warning.

---

**Quoted vs unquoted heredoc — variable expansion test:**

```bash
ANALYST="aba Mensah"

# Unquoted delimiter — shell expands $ANALYST
cat << EOF
Prepared by: $ANALYST
EOF
# Output: Prepared by: aba Mensah

# Quoted delimiter — shell treats $ANALYST as a literal string
cat << 'EOF'
Prepared by: $ANALYST
EOF
# Output: Prepared by: $ANALYST
```

**What is the difference?**

With an unquoted delimiter, the heredoc body undergoes the same expansions as a double-quoted string: `$VARIABLE` is replaced with its value, `$(command)` runs command substitution, and `` `backtick commands` `` are executed. With a quoted delimiter (single or double quotes around the word), no expansion of any kind occurs. The body is treated as a raw string.

**When to use each in a real script:**

- **Unquoted heredoc** use when you want to generate output dynamically: building a configuration file from variables, generating an email body that includes a hostname or timestamp, or writing a script that references current environment values.

  ```bash
  # Generate nginx config from variables
  SERVER_NAME=cyphercore.io
  cat > /etc/nginx/sites-available/app.conf << EOF
  server {
      server_name $SERVER_NAME;
      listen 443 ssl;
  }
  EOF
  ```

- **Quoted heredoc** use when the output must be treated as literal text: writing a shell script that contains its own `$VARIABLES` (which should not be expanded now), writing documentation, or writing any content where unexpected expansion could corrupt the output.

  ```bash
  # Write a template script $PORT must appear literally in the output
  cat > /usr/local/bin/start_service.sh << 'EOF'
  #!/bin/bash
  echo "Starting on port $PORT"  # This $PORT will expand when the script runs, not now
  EOF
  ```
